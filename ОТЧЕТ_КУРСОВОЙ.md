## 1. Термины и определения

- **Frontend** — клиентская часть приложения (Angular).
- **Backend** — серверная часть приложения (Node.js, Express).
- **MongoDB** — база данных для хранения информации о пользователях, событиях и регистрациях.
- **Docker** — система контейнеризации.
- **Kubernetes (Minikube)** — система оркестрации контейнеров.
- **CI/CD** — процессы автоматизации сборки, тестирования и деплоя.

## 2. Перечень сокращений и обозначений

- **API** — Application Programming Interface
- **CRUD** — Create, Read, Update, Delete
- **JWT** — JSON Web Token
- **HPA** — Horizontal Pod Autoscaler
- **VPS** — Virtual Private Server

## 3. Введение

Цель проекта — разработка платформы для управления мероприятиями с регистрацией пользователей, хранением данных, автоматизацией деплоя и мониторинга. Проект реализован с использованием современных технологий: Angular, Node.js, MongoDB, Docker, Kubernetes.

## 4. Основная часть

### 1. Анализ предметной области

- Система предназначена для организации и управления событиями (конференции, вебинары, мастер-классы).
- Пользователи могут регистрироваться, просматривать события, управлять профилем.
- Администраторы — создавать и редактировать события, просматривать регистрации.

#### 1.1. Архитектура проекта

- Многоуровневая архитектура: frontend, backend, БД, инфраструктура.
- Используется REST API для взаимодействия между слоями.
- Контейнеризация всех компонентов.

#### 1.2. Структура проекта

```
курсовая_работа/
├── backend/         # Node.js + Express
├── frontend/        # Angular
├── k8s/             # Kubernetes манифесты
├── docker-compose.yml
├── README.md
└── ...
```

#### 1.3. Модели данных

- **User**: email, пароль, роль, профиль
- **Event**: название, описание, дата, организатор
- **Registration**: пользователь, событие, статус

### 2. Выбор и обоснование технологий

#### 2.1. Используемые технологии

- **Angular** — современный фреймворк для SPA
- **Node.js + Express** — быстрый backend
- **MongoDB** — NoSQL БД
- **Docker** — контейнеризация
- **Kubernetes (Minikube)** — оркестрация
- **Nginx** — веб-сервер и прокси
- **Prometheus + Grafana** — мониторинг
- **GitHub Actions** — CI/CD

#### 2.2. Обоснование выбора

- Высокая масштабируемость
- Простота деплоя и поддержки
- Возможность горизонтального масштабирования (HPA)
- Автоматизация тестирования и доставки

#### 2.3. Сравнительный анализ

- Docker Compose — для локальной разработки
- Kubernetes — для продакшн и масштабирования
- MongoDB vs. PostgreSQL — выбрана MongoDB за простоту и гибкость

#### 2.4. Преимущества стека

- Быстрая разработка
- Легкая миграция между средами
- Автоматизация процессов

### 3. Практическая реализация

#### 3.1. Организация контейнерной инфраструктуры

- Все сервисы упакованы в Docker-образы
- Используется docker-compose для локального запуска
- Kubernetes манифесты для продакшн

#### 3.2. Реализация backend

- Node.js + Express
- REST API: /api/events, /api/users, /api/registrations
- JWT-аутентификация
- Валидация данных
- Логирование запросов (morgan)

#### 3.3. Реализация хранения данных

- MongoDB, коллекции: users, events, registrations
- Индексация по email, id событий
- Миграции и seed-скрипты для тестовых данных

#### 3.4. Реализация frontend

- Angular 14
- Модульная структура: auth, events, admin, profile
- Адаптивный дизайн
- Валидация форм
- Интеграция с REST API

#### 3.5. Конфигурация Nginx

- Проксирование /api на backend
- Сервинг статических файлов
- Конфигурируемый через ConfigMap в Kubernetes

#### 3.6. Мониторинг и логирование

- Prometheus: сбор метрик
- Grafana: визуализация
- Логирование через morgan и kubectl logs

#### 3.7. Кластерная архитектура

- Minikube: локальный кластер
- HPA: автоматическое масштабирование по CPU/RAM
- Namespace: изоляция ресурсов

#### 3.8. CI/CD

- GitHub Actions: автоматическая сборка и тесты
- Автодеплой в Minikube/VPS
- Проверка метрик после деплоя

#### 3.9. Деплой на сервер

- Сборка Docker-образов
- Загрузка в кластер
- Применение манифестов
- Проверка статуса pods и сервисов

### Метрики и тесты

- **Метрики производительности**: время отклика API, загрузка CPU/RAM, время деплоя
- **Тесты**:
  - Unit-тесты (Jest, Jasmine)
  - E2E-тесты (Cypress)
  - Проверка API через Postman
  - Проверка UI через Selenium
- **Покрытие тестами**: >80% для основных модулей
- **Мониторинг**: Grafana dashboard, alerts
